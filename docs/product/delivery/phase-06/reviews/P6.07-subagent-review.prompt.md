### Attack surfaces to probe

- `+layout.svelte` `$effect` for store sync: `session.set(data.session)` etc. — does `$effect` correctly track `data.session`, `data.profile`, `data.projects` as reactive reads when `data` comes from `$props()`? (Store reads inside `$effect` are valid, but confirm the prop-derived fields are tracked.)
- `+layout.svelte` `onMount` uses `data.supabase` — captures the value at mount time. If `supabase` changes between mounts (page reload), there's no re-subscription. Verify this is the same behavior as the original `export let data` + `$: ({ supabase } = data)` pattern (original also captured `supabase` at the `$:` destructuring level, not in `onMount` directly).
- `+layout.svelte` second `$effect` (URL redirect): reads `$selectedRange` and `$profile` (stores) AND `window.location.href`. In Svelte 5, `window.location.href` is not reactive — the effect only re-runs when store values change, not when the URL changes. Is this the same behavior as the Svelte 4 `$:` statement (which also only re-ran when store values changed)?
- `+page.svelte` `summariesOverride` type: `$state<typeof data.summaries | null>(null)` — when `data` changes on navigation, `summariesOverride` retains its non-null value from the previous fetch. Does the `$derived(summariesOverride ?? data.summaries)` correctly prefer fresh `data.summaries` after navigation? (It does not reset `summariesOverride` on data change — same as the original `let summariesOverride: typeof data.summaries | null = null` behavior.)
- `SearchInput.svelte` `search` sync: `$effect(() => { search = query })` syncs `search` from `query` prop. The `eslint-disable` is present. Verify that `bind:value={search}` still works correctly — `search` is writable `$state` and `bind:value` on a native input uses the value as its source of truth.
- `login-redirect/+page.svelte` reactive redirect: `$effect(() => { ... location.href = ... })` — in Svelte 5, `$effect` runs after mount (not synchronously during render). The redirect fires after the DOM is mounted. This is the same semantic as the Svelte 4 `$: { ... }` reactive statement which also ran asynchronously after reactive updates. Confirm there's no SSR issue (the `location.href` assignment is browser-only).
- `projects/+page.svelte` uses `data.supabase` directly in handler functions — not destructured ahead of time. If `data.supabase` is updated on navigation, future calls to `onTrackProject` etc. would use the new `supabase` instance. This is better than the original which captured `supabase` in `$: ({ ..., supabase } = data)` — confirm the direct access is equivalent.
