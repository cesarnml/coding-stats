`DailyGauge` correctness during initial mount depends on the Svelte 5 guarantee that `$effect` callbacks fire in declaration order during the mount flush (so the selectedDate effect settles `selectedDate = defaultDate` before `onMount` reads `option`). This guarantee holds under the current file layout. If a future refactor moves the `onMount` block above the selectedDate `$effect`, the gauge would initialize ECharts with `data[0].value = undefined` for one frame before correcting. The dependency on declaration order is not documented. A comment or a restructuring that initializes `selectedDate` synchronously (e.g., `let selectedDate = $state(availableDates.at(-1) ?? '')`) would eliminate the ordering dependency entirely — though that would require the `availableDates` derivation to be declared above `selectedDate`, which may conflict with how `$props()` and `$derived` interact on the initial computation. This is lower priority than the ticket-doc gap above; noting it for T05 or a future audit.

`Treemap` passes `$page.params.projectName` (type: `string | undefined`) into `createProjectFileToTimeDict` and `createTreemapOption` which declare `projectName: string`. If the component ever renders outside a route that provides `projectName` (e.g., a test or a route refactor), the helpers will silently split on the string `"undefined"` and produce garbage tree keys rather than throwing. This is a pre-existing gap untouched by this migration; flagging for visibility since the runes migration is an opportunity to audit it.
