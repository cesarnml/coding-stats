**Actionable findings**

**Finding 1 — `onClick` in `DailyChartControls` has no error handling; `loading` gets stuck on network failure.**

`DailyChartControls.svelte:69–79`. The `onClick` async function is:

```js
const onClick = async (step: Step) => {
  loading = true
  ...
  const response = await fetch(...)  // can throw
  durations = await response.json()  // can throw
  onupdate?.(durations)
  loading = false                    // never reached on throw
}
```

If either `await` rejects, the function exits without resetting `loading = false`. Both navigation buttons have `disabled={isPrevDisabled}` and `disabled={isNextDisabled}` which derive from `loading` (among other conditions). Because `loading` stays `true`, the buttons remain disabled until page reload. The `setInterval` callbacks at lines 41–63 DO have try/catch with `loading = false` outside the try — the inconsistency is notable and the gap is observable by the user. A user on a flaky connection clicking Prev/Next gets their controls silently frozen. Fixing requires wrapping `onClick` body in try/finally with `loading = false` in the `finally`.

**Advisory Observations**

- **Route pages still use Svelte 4 idioms (`export let`, `$:`).** Both `src/routes/+page.svelte` and `src/routes/projects/[projectName]/+page.svelte` were touched for the `onwakarange` call-site update, but they still use `export let data`, `$: summaries = ...`, and other legacy reactive syntax throughout. Svelte 5 compiles both modes side-by-side without issue, but when those pages are eventually migrated, the legacy `$:` blocks will need to become `$derived` and `$effect` calls, and the handler signatures that still reference `CustomEvent<T>` patterns in adjacent code should be audited again.

- **`DailyChartControls` local `durations` does not track prop changes after mount.** `let durations = $state(durationsProp)` captures the initial prop value only. If the parent (`ActivityChart`, `TimelineChart`) ever receives a new `durations` prop from its own parent (e.g., after a page-level `invalidate`), `DailyChartControls`' local state is not updated. The component's internal controls would continue showing the stale date. In the current architecture this appears to be fine — the daily data panel is self-refreshing and the route-level `durations` prop originates from SSR load data that doesn't change in-session — but it is a silent semantic difference from Svelte 4's `export let`, which did track prop reassignments reactively.

Given the core behavioral invariant of this ticket is "callbacks fire where dispatches did," both specs leave the primary migrated behavior untested.
