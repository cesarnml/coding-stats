### Findings

#### F1 — MEDIUM | `$effect` reactive granularity is coarser than the Svelte 4 analog

**File:** `Navbar.svelte:15-19`

```svelte
$effect(() => {
  if ($media.sm) {
    dropdown.close()
  }
})
```

`$media` is a `readable` store that emits a **new `MediaBreakpoints` object** whenever any of its five breakpoints change via `matchMedia` listeners. The `$effect` auto-subscribes to `$media` as a whole — not to `$media.sm` specifically. This means the effect re-runs on every breakpoint transition (including `md→lg`, `lg→xl`, etc.), not only when `sm` changes.

At each re-run, if `$media.sm` is already `true` (viewport has been above 640px since last resize event), `dropdown.close()` is called redundantly. In isolation this is benign — `dropdown.close()` sets the store to `false` idempotently. However, if any resize event fires while the dropdown is intentionally open on a non-`sm` breakpoint toggle (e.g., user at 800px triggers `md` crossing), the effect fires, `$media.sm` is still `true`, and the dropdown is silently closed. This is the **same behavior as the Svelte 4 `$: if ($media.sm)` reactive statement** (which also re-ran on all `$media` changes), so it is not a regression introduced by this ticket — but it is the pre-existing behavior now made explicit.

**Verdict:** Not a regression. Documenting because the attack surface question is answered: the coarser granularity is inherited from the Svelte 4 code and does not introduce new failure paths.

#### F2 — LOW | Spec guard does not exercise any of the ticket's behavioral invariants

**File:** `Navbar.spec.ts:1-12`

The existing spec renders `Navbar` and asserts a `<nav>` element and logo image exist. It does not exercise:

- Dropdown open/close toggle via `dropdown.open()`/`dropdown.close()`
- `$effect` firing on `$media.sm` to close the dropdown
- `$derived` recomputing `navUrls`/`dropdownUrls` when `$session` changes
- The snippet passthrough rendering `<img>` inside `NavLogo`'s button

If the runes migration silently broke store auto-subscription inside `$derived` or `$effect`, this spec would not catch it. The test as written would pass even if all reactive behavior was dead. This is a pre-existing gap, not introduced here, but the ticket's smoke assertion ("mobile nav dropdown opens/closes; loading/navigating indicator still reacts") has **no automated guard** — only manual verification covers it.

**Verdict:** Advisory observation. No ticket-scope action required since `Red: skip` was the decision, but worth noting for a follow-on hardening ticket.

#### F3 — LOW | `NavLinks.svelte` label silently empty for Home URL

**File:** `NavLinks.svelte:34`

```svelte
<NavLink {url} label={url.split('/')[1]} />
```

`Url.Home` is `'/'`. `'/'.split('/')` produces `['', '']`, so `[1]` is `''` — an empty string. `NavDropdown.svelte:25` has a `|| 'Home'` fallback for exactly this case; `NavLinks.svelte` does not. If `Url.Home` appears in `TOP_LEVEL_NAV_URLS`, the top-level nav link for it renders a button with no visible label. This is **pre-existing and not introduced by this ticket**. Included here because the diff touched this line and the inconsistency between the two files is now visible in the same PR.

**Verdict:** Advisory observation. Pre-existing gap.
